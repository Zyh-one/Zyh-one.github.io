## 2. 大模型推理系统到底在优化什么

推理系统优化通常围绕 4 个核心目标展开：

1. 降低单请求延迟
2. 提高整体吞吐
3. 降低显存与带宽压力
4. 在满足 SLO 的前提下提高 goodput

这里要先区分 4 个概念：

- `latency`：单个请求从发起到完成的耗时
- `throughput`：单位时间内处理的请求数或 token 数
- `SLO`：系统对 TTFT、TPOT、尾延迟等指标的目标约束
- `goodput`：在满足 SLO 的前提下真正“有效完成”的吞吐

为什么推理优化和普通深度学习推理不一样：

- LLM 是自回归生成，token 是一个一个产出的
- 同一个请求会经历 `prefill` 和 `decode` 两种完全不同的计算阶段
- 长上下文会引入巨大的 KV Cache
- 多请求并发时，系统既要考虑算力，也要考虑 cache、调度、公平性和尾延迟

所以大模型推理系统本质上是“模型计算 + 内存系统 + 调度系统 + 服务系统”的组合问题。

## 3. Transformer 推理里的关键计算结构

### 3.1 为什么 attention 是推理的核心瓶颈

自回归模型生成时，每一步都要基于历史上下文计算当前 token 的表示。对第 `t` 个 token 来说：

- `Q` 只和当前 token 有关
- `K`、`V` 需要覆盖历史 token
- 如果每步都重算历史 `K/V`，成本会很高

因此推理系统几乎都会引入 `KV Cache`：

- 历史 token 的 `K/V` 存起来
- 生成下一个 token 时只算新的 `Q/K/V`
- 用新的 `Q` 去和缓存中的历史 `K/V` 做 attention

结论：

- 没有 KV Cache，decode 会非常慢
- 有了 KV Cache，算力压力下降，但显存压力会上升

这就是为什么很多推理优化最后都会落到“如何管理 KV Cache”。

### 3.2 MHA、MQA、GQA 为什么重要

推理岗位很爱问 attention 结构差异，因为它直接影响 KV Cache 大小和带宽压力。

#### MHA

- 每个 query head 都有自己独立的 key/value head
- 表达能力强，但 KV Cache 更大

#### MQA

- 多个 query head 共享同一组 key/value
- KV Cache 显著缩小
- 更省显存和带宽，但表达能力可能有一定折中

#### GQA

- 介于 MHA 和 MQA 之间
- 多个 query head 分组共享 key/value
- 兼顾效果和效率

面试中可以这么总结：

`MHA -> 表达能力优先，MQA -> 推理效率优先，GQA -> 二者折中。`

## 4. Prefill 与 Decode

### 4.1 两个阶段为什么要分开理解

LLM 推理不是单一过程，而是两段式：

#### Prefill

- 处理完整 prompt
- 一次性把上下文编码进模型
- 计算量大，通常更偏 compute-bound
- 会写入大量 KV Cache

#### Decode

- 基于已有上下文逐 token 生成
- 每次只处理一个或少量 token
- 更容易受 memory-bound、调度和 cache 访问影响
- 用户感知很强，因为它决定输出流畅度

面试时最重要的表达：

- `Prefill` 决定 TTFT 很大一部分表现
- `Decode` 更直接影响 TPOT、ITL 和流式体验

### 4.2 为什么 Prefill / Decode 分离越来越重要

因为这两阶段的资源偏好不一样：

- Prefill 更吃大吞吐和大批量计算
- Decode 更吃低延迟、稳定 cache 和高频小步调度

如果强行混在一起跑，会出现：

- 新进来的长 prompt 抢占算力
- 已经在 decode 的请求 inter-token latency 变差
- 尾延迟恶化

因此现代推理系统会做几类优化：

- 在调度器里显式区分 prefill 与 decode
- 做 `chunked prefill`
- 做 `disaggregated prefill`

## 5. KV Cache、Paged Attention 与 Cache 管理

### 5.1 KV Cache 为什么是推理系统的“内存核心”

KV Cache 的收益：

- 避免重复计算历史 token 的 `K/V`
- 大幅提升 decode 速度

KV Cache 的代价：

- 上下文越长，占用显存越多
- 并发请求越多，总 cache 越大
- cache 访问和搬运会变成性能瓶颈

因此推理系统通常要回答 5 个问题：

1. cache 怎么分配
2. cache 怎么复用
3. cache 怎么回收
4. cache 满了怎么办
5. cache 如何和调度器联动

### 5.2 为什么 Paged Attention 很关键

Paged Attention 的核心思想是：

- 不把每个请求的 KV Cache 看成一整块连续大内存
- 而是拆成很多 block / page
- 用 block table 管理逻辑序列到物理块的映射

好处：

- 降低连续大块显存分配难度
- 更方便做动态扩缩
- 更容易支持前缀复用与 cache 管理
- 更适合多请求并发场景

你可以把它类比成操作系统的分页内存管理。

面试里建议这样说：

`Paged Attention` 不只是一个 attention kernel 名字，它背后是把 KV Cache 组织方式从“连续分配”改成“按块管理”，这样才能更好支持大规模在线推理。

### 5.3 cache 复用的常见方向

常见复用思路包括：

- prefix caching
- block 级复用
- 共享 prompt 的 KV 复用
- disaggregated serving 场景下的 KV transfer

但要注意：

- 复用越激进，系统复杂度越高
- 一旦引入跨请求 KV 共享，正确性、生命周期和失效管理都会更复杂

## 6. Continuous Batching 与调度器

### 6.1 Continuous Batching 是什么

它也常被叫做：

- in-flight batching
- iteration-level batching

核心不是简单把多个请求拼成一个 batch，而是：

- 在每一轮迭代里动态决定哪些请求进入当前 batch
- 有些请求可能在 prefill
- 有些请求可能在 decode
- 请求会持续进入和退出批次

目标：

- 提高 GPU 利用率
- 降低空转
- 在不明显恶化延迟的前提下提高吞吐

### 6.2 为什么它比静态 batching 更适合 LLM

静态 batching 的问题：

- 请求长度不同
- 输出长度不同
- 一个 batch 内很容易出现 padding 浪费
- 长请求会拖慢短请求

Continuous batching 的优势：

- batch 是流动的，不是固定的一次性集合
- 能把处在不同阶段的请求更灵活地组织起来
- 更适合真实在线流量

### 6.3 调度器通常关心什么

一个 LLM 推理调度器通常要同时考虑：

- 当前等待队列里有哪些请求
- 哪些请求已经在运行
- 还有多少 KV Cache block 可用
- 当前 batch token 数是否超限
- 当前 batch request 数是否超限
- 是否需要 preempt 某些请求
- 是否优先 decode 以保证流畅度
- 是否允许把长 prompt 切块处理

面试时可以这么总结：

`调度器本质是在吞吐、显存、延迟、公平性之间做多目标平衡。`

### 6.4 chunked prefill 是什么

长 prompt 的 prefill 很容易把 decode 请求“顶出去”。因此一个常见办法是：

- 不一次性把超长 prompt 全部 prefill 完
- 而是切成若干 chunk
- 让调度器在 chunk 之间穿插 decode

好处：

- 缓解长 prompt 对 decode 尾延迟的冲击

代价：

- chunk size 不好选
- 太小会增加调度开销
- 太大又无法解决尾延迟问题

## 7. 并行策略

这部分是推理岗位高频基础题，建议你按“为什么需要、怎么切、代价是什么”来答。

### 7.1 Data Parallel

定义：

- 模型完整复制到多个 GPU 或实例上
- 各副本处理不同请求

优点：

- 最简单
- 易扩展
- 对吞吐提升直接

缺点：

- 显存占用高，因为权重复制多份
- 不解决单模型放不下单卡的问题

适用场景：

- 模型能放进单卡
- 想横向扩容吞吐

### 7.2 Tensor Parallel

定义：

- 把单层里的张量计算拆到多个 GPU 上
- 例如线性层、attention 投影等按列或按行切分

优点：

- 能让单卡装不下的模型跑起来
- 对大模型推理很常见

缺点：

- GPU 间通信频繁
- 推理时每步 decode 都会受通信影响

适用场景：

- 单机多卡
- 模型太大，单卡放不下

### 7.3 Pipeline Parallel

定义：

- 按层切模型
- 不同 stage 放在不同 GPU 或节点上

优点：

- 适合模型极大、跨节点部署
- 减少某些层内切分的通信压力

缺点：

- 会引入流水线气泡
- 在线推理中单请求低 batch 情况下不一定理想
- stage 间负载不平衡会影响性能

适用场景：

- 多机多卡
- 模型规模极大

### 7.4 Expert Parallel

主要对应 MoE 模型。

定义：

- 不同专家放到不同 GPU
- router 为 token 选择少量专家执行

优点：

- 非常适合 MoE 的稀疏激活特性
- 能利用专家天然并行性

缺点：

- token dispatch / gather 复杂
- 负载均衡难
- 热门专家可能成为瓶颈

### 7.5 常见组合思路

真实系统里经常不是单一策略，而是组合：

- `DP + TP`
- `TP + PP`
- `DP + EP`
- attention 层走 DP，MoE expert 层走 EP 或 TP

面试里最稳的回答不是背定义，而是说：

- `DP` 主要扩吞吐
- `TP` 主要解决单卡放不下
- `PP` 主要解决跨节点超大模型
- `EP` 主要服务 MoE
- 组合策略的核心取舍在于显存、通信和在线延迟

## 8. 模型架构与推理性能的关系

### 8.1 Dense vs MoE

#### Dense

- 每个 token 都走全部参数
- 路径简单，部署和调度更直接

#### MoE

- 每个 token 只激活少数专家
- 理论上可以在增大总参数的同时保持较低激活成本

但 MoE 推理并不天然更简单，因为它会引入：

- router 决策
- token 到专家的分发
- 专家负载不均
- all-to-all 通信

所以面试里不要只说“MoE 更省算力”，要补一句：

`MoE 把计算问题部分转成了通信、路由和负载均衡问题。`

### 8.2 长上下文相关结构

长上下文推理里常见关注点：

- sliding window attention
- chunked attention
- 更小的 KV Cache 设计
- KV cache offloading / quantization

你可以记一个主线：

长上下文不是只考模型能不能看到更多 token，更考验 cache 能不能存下、带宽能不能扛住、调度能不能稳定。

### 8.3 FlashAttention 的位置

FlashAttention 的核心是 IO-aware attention 优化。

它的价值在于：

- 通过更好的显存访问与 tile 策略减少 HBM 读写
- 在不改变数学结果的前提下提升 attention 效率

面试时你可以说：

`FlashAttention` 更偏 kernel / 算子层优化，而 `Paged Attention` 更偏 KV Cache 组织与在线推理系统优化。二者都重要，但解决的问题不完全一样。

## 9. 量化

### 9.1 为什么量化是推理基础能力

量化最直接的目标：

- 降低显存占用
- 降低带宽压力
- 提升可部署性

但量化不一定总能无脑提速，因为：

- 有些方案更省内存但 kernel 不够快
- 有些方案精度损失更敏感
- 有些方案需要校准数据

所以量化的本质不是“位宽越低越好”，而是：

`在精度、速度、显存之间找到可接受平衡。`

### 9.2 常见量化类型

#### FP16 / BF16

- 现在最常见的高精度推理基线
- 工程成熟
- 效果稳定

#### INT8

- 显存和带宽更省
- 常作为比较稳健的压缩方案

#### INT4 / W4A16 一类 4bit 方案

- 压缩比更高
- 很适合部署更大模型
- 但更依赖量化算法和 kernel 支持

#### FP8

- 近来很重要
- 在支持良好的硬件和内核上，能在效率与精度之间提供较好折中

### 9.3 常见量化方法

#### bitsandbytes

- 工程上很常用
- 容易加载 8bit / 4bit 模型
- 上手快

#### GPTQ

- 后训练量化代表方案之一
- 常见于 4bit 权重量化
- 一般需要校准数据

#### AWQ

- 强调 activation-aware
- 通过保护关键权重来减小性能损失

#### Quanto / torchao / TensorRT-LLM recipes

- 越来越多框架提供统一量化后端或多种 recipe
- 重点不只是压缩，还包括 kernel 适配和端到端可用性

### 9.4 量化在面试里怎么讲比较稳

建议你说清 3 层：

1. `weight-only quantization`

- 主要压缩权重
- 实现简单，部署常见

2. `activation quantization`

- 进一步压缩激活
- 难度更高，对 kernel 和校准更敏感

3. `KV cache quantization`

- 主要针对长上下文和高并发场景
- 因为很多时候不是权重先爆，而是 KV Cache 先爆

这会显得你不是把量化只理解成“加载一个 4bit 模型”。

## 10. 推理服务指标

### 10.1 必须会说的指标

#### TTFT

- `Time To First Token`
- 从请求发出到第一个有效 token 返回的时间
- 对 chat、agent、交互式系统体验影响极大

#### ITL / TPOT

- `Inter-Token Latency`
- `Time Per Output Token`
- 衡量后续 token 的生成速度与流畅度

#### End-to-End Latency

- 请求完整结束的总耗时

#### Request Throughput

- 每秒处理多少请求

#### Token Throughput

- 每秒处理多少输入 token / 输出 token

#### P50 / P95 / P99

- 中位数和尾延迟
- 真实服务里尾延迟常比平均值更重要

### 10.2 为什么 TTFT 和 TPOT 要分开看

因为它们受不同因素影响：

- TTFT 更受 prompt 长度、prefill 负载、排队和调度影响
- TPOT / ITL 更受 decode 阶段、KV Cache、kernel、通信和 batch 组织影响

一个系统可能：

- TTFT 很好，但 TPOT 很差
- 或者吞吐很高，但尾 ITL 很差

所以只说“tokens/s 很高”在面试里通常不够。

### 10.3 goodput 为什么越来越常见

因为高吞吐不代表用户体验真的好。

例子：

- 系统吞吐很高
- 但很多请求超出 TTFT 或 TPOT SLO
- 那这些吞吐对业务其实价值有限

因此很多近年的系统论文开始强调：

- SLO attainment
- goodput
- heterogenous SLO-aware scheduling

这也是为什么 PD 分离、SLO-aware scheduling 越来越重要。

## 11. PD 分离与 PD Disaggregation

### 11.1 概念区分

这里有两个层次：

#### 逻辑上的 Prefill / Decode 分离

- 在一个服务实例内部区分两阶段
- 调度器知道哪些请求在 prefill，哪些在 decode

#### 物理上的 PD Disaggregation

- 把 prefill 和 decode 放到不同实例、不同 GPU、甚至不同节点
- prefill 实例把 KV Cache 传给 decode 实例

### 11.2 为什么要做 PD Disaggregation

主要动机有两个：

1. 分别优化 TTFT 和 ITL

- prefill 和 decode 资源偏好不同
- 分开后可以给两端配置不同并行策略

2. 控制尾延迟

- 避免重 prefill 任务干扰 decode 请求
- 尤其适合交互式和流式场景

### 11.3 代价是什么

PD 分离不是白送的，它会引入：

- KV cache 传输成本
- 实例间通信成本
- 更复杂的调度和故障处理
- 额外的系统架构复杂度

因此面试里建议你不要把它说成“绝对更好”，而要说：

`PD Disaggregation 主要是为了更好地分别优化 TTFT 与 ITL、控制尾延迟，但是否划算取决于 KV 传输成本和业务流量形态。`

### 11.4 什么时候特别值得考虑

比较适合：

- 长 prompt、多轮对话
- 对流式体验敏感
- 混合 workload 很复杂
- 希望把 prefill 资源池和 decode 资源池独立扩缩

## 12. 线上推理服务的常见优化方向

### 12.1 模型层

- 换更高效的 attention 结构
- 使用 GQA / MQA
- MoE 稀疏化
- 更适合推理的模型变体

### 12.2 算子层

- FlashAttention
- fused kernel
- CUDA Graph
- 更高效的量化 kernel

### 12.3 内存层

- Paged Attention
- KV Cache 压缩
- prefix caching
- KV offloading

### 12.4 调度层

- continuous batching
- chunked prefill
- preemption
- SLO-aware scheduling
- PD disaggregation

### 12.5 服务层

- data parallel 扩缩容
- 路由与负载均衡
- 多租户隔离
- 指标监控与 autoscaling

## 13. 面试里最容易被追问的点

### 13.1 “为什么 decode 常常比 prefill 更难优化体验”

建议答法：

因为 prefill 更像大块计算，重点是吞吐；decode 是高频小步迭代，更依赖 KV Cache、调度和低尾延迟控制。用户会持续感知 decode 阶段的抖动，所以它对系统设计更敏感。

### 13.2 “为什么 continuous batching 能提升吞吐”

建议答法：

因为在线请求长度不一致，静态 batch 很浪费。continuous batching 让请求按迭代动态进出批次，减少 padding 和空转，提高 GPU 利用率。

### 13.3 “为什么高吞吐不一定代表服务好”

建议答法：

因为用户体验更看 TTFT、TPOT 和尾延迟。如果吞吐很高但大量请求超出 SLO，实际 goodput 并不高。

### 13.4 “为什么 PD 分离不是所有场景都用”

建议答法：

因为它会带来 KV transfer 和更复杂的系统编排。只有当 TTFT / ITL 的独立优化价值明显大于通信与复杂度成本时，它才真正划算。

### 13.5 “量化是不是越低比特越好”

建议答法：

不是。量化要同时看精度损失、kernel 支持、内存节省和端到端延迟。有些低比特方案显存很省，但真实服务速度不一定更好。

## 14. 你可以主动输出的结构化表达

如果面试官让你概括推理系统，你可以用这套表达：

大模型推理优化通常分 5 层：

1. 模型结构层：比如 MHA/MQA/GQA、Dense/MoE、长上下文结构
2. 算子层：比如 FlashAttention、fused kernel、CUDA Graph
3. 内存层：比如 KV Cache、Paged Attention、KV 量化与复用
4. 调度层：比如 continuous batching、chunked prefill、PD 分离
5. 服务层：比如并行扩展、负载均衡、SLO 与 goodput

如果面试官问“你最关注哪一层”，你可以接：

我最关注的是内存层和调度层，因为大模型在线推理很多时候不是纯算力瓶颈，而是 KV Cache 管理和请求调度问题。
